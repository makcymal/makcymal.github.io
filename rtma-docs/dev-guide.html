    Разработчикам

    Данный материал описывает подробности реализации системы мониторинга и предназачен для разработчиков, занимающихся дальнейшим развитием системы.

    Общий обзор

    Три компонента составляют основу системы:

    1. Сенсор – программа, исполняемая на множестве вычислительных узлов (физических или виртуальных), собирающая данные об объемах сетевого трафика и использовании аппаратных ресурсов.

    2. Клиент – программа, запрашивающая и получающая мониторинговые данные с сенсоров. Частным случаем клиента является

      • Веб-приложение – клиент, позволяющий просматривать мониторинговые данные в браузере.

    3. Сервер – программа, исполняемая на одном устройстве, непрерывно собирающая данные с сенсоров, обрабатывающая и агрегирующая их и отправляющая по запросу в веб-приложение.

    Взаимодействие всех компонентов – сенсоров, сервера и клиентов, можно описать следующей схемой:

    Клиенты никогда не взаимодействуют с сенсорами напрямую, все взаимодействие происходит посредством сервера между ними.

    Для удобства работы с сенсорами они имеют следующие свойства:

    • sensor_id – идентификатор, привязанный к конкретной машине, на которой запущен сенсор. Он не изменяется и является уникальным среди всех сенсоров (если только никакие два сенсора не запущены на одной машине).
    • sensor_label и sensor_group – метка и группа сенсора, не привязано к конкретной машине, устанавливается изначально в файле конфигурации сенсора и может быть изменено (при этом файл конфигурации перезаписывается для сохранения изменений). Метка должна быть уникальной в рамках одной группы (в разных группах два сенсора могут иметь одинаковую метку). Метка и группа создают двухступенчатую иерархию сенсоров, где каждый сенсор принадлежит сначала какой-то группе, а затем внутри нее обладает своей меткой. Это позволяет производить мониторинг целых групп сенсоров. При изменении группы сенсора считается, что он перемещается в другую группу (даже если это новая группа).

    Кратко взаимодействие компонентов можно описать следующим образом:

    1. Клиент формирует запрос к серверу с тем, какие мониторинговые данные он хочет получать, и от какого сенсора (или группы сенсоров).

    2. Сервер для каждого сенсора агрегирует запросы к нему от всех клиентов в один запрос и отправляет его сенсору.

    3. Сервер подписывается на получение мониторинговых данных сенсора с этим агрегированным запросом.

    4. Сенсор начинает собирать мониторинговые данные (то, что он собирает, определяется запросом от сервера) и отправлять серверу.

    5. Сервер разбирает пришедшие мониторинговые данные и отправляет клиентам то, что они запрашивали (и ничего сверх того).

    6. При отключении последнего клиента, запрашивающего мониторинг некоторого сенсора, сервер отписывается от этого сенсора, и тот приостанавливает свою работу до следующей подписки.

    Сенсоры работают по принципу on-demand: мониторинговые данные собираются и передаются только в случае, если есть запрос со стороны сервера. Такой запрос к конкретному сенсору у сервера появляется по двум причинам: подключился клиент и хочет просмотреть данные с одного сенсора или группы сенсоров, или сервер по своей воле собирает данные с сенсора, чтобы в любой момент по запросу клиента он мог сформировать статистику за последние 24 часа. Такой подход снижает расходы на трафик и вычислительные ресурсы для передачи мониторинговых данных, хотя и усложняет устройство всей системы.

    Принятая терминология

    Данные термины в том или ином виде встречаются в исходном коде и требуются для того, чтобы систематизировать наименование конкретных сущностей в коде, комментариях, документации и среди разработчиков.

    • metric (метрика) – наименование показателя, данные по которому может собирать сенсор. Например: "freq" (частота процессора) или "cached" (объем кешей).

    • measurement (измерение) – значение, которое сенсор ставит в соответствие конкретной метрике.

    • rubric (рубрика) – несколько логически связанных метрик.

      Список рубрик и метрик:
      cpu – метрики, относящиеся к загруженности процессора:
      • system: процессорное время, затраченное процессами, выполняющимися в kernel mode,
      • user: процессорное время, затраченное процессами, выполняющимися в user mode,
      • nice: процессорное время, затраченное приоритетными процессами, выполняющимися в user mode,
      • idle: процессорное время простоя,
      • iowait: процессорное время, проведенное в ожидании завершения операций ввода-вывода,
      • irq: процессорное время, затраченное на обслуживание аппаратных прерываний,
      • softirq: процессорное время, затраченное на обслуживание программных прерываний,
      • steal: процессорное время, проведенное в виртуализированной среде,
      • guest: процессорное время, затраченное на обслуживание виртуального процессора для гостевой ОС,
      • guest_nice: процессорное время, затраченное на обслуживание виртуального процессора для приоритетных процессов гостевой ОС,
      • freq: частота процессора.
      net – метрики, относящиеся к сетевому трафику:
      • recv: объем полученных данных,
      • sent: объем отправленных данных,
      • errin: количество ошибок при получении данных,
      • errout: количество ошибок при отправке данных,
      • dropin: количество пакетов, отброшенных при получении данных,
      • dropout: количество пакетов, отброшенных при отправке данных.
      mem – метрики, относящиеся к использованию оперативной памяти:
      • used: объем используемой памяти,
      • cached: объем оперативной памяти, используемой для кешей,
      • shared: объем оперативной памяти, общей для нескольких процессов,
      • swap: объем используемого раздела swap.
      dsk – метрики, относящиеся к использованию постоянной памяти:
      • used: объем используемой постоянной памяти,
      • read: объем данных, прочтенных из постоянной памяти,
      • write: объем данных, записанных в постоянную память.

    • prompt (промпт) – запрос для сенсора со списком метрик, которые ему необходимо собирать и отправлять. Перечисляет метрики, которые будет собирать сенсор, а также интервал времени в секундах, в течение которого сенсор будет спать до следующего сбора и отправки измерений.

    • unit – единица измерения каждой отдельной метрики. Одна метрикa может иметь разную единицу измерения в разных промптах, например, mem.used может быть выражен в килобайтах или в процентах.

      Единицы измерения по умолчанию:
      cpu system user nice idle iowait irq softirq steal guest guest_nice freq
      Unit % % % % % % % % % % mhz
      net recv sent errin errout dropin dropout
      Unit kb kb pcs pcs pcs pcs
      mem used cached shared swap
      Unit kb kb kb kb
      dsk used read write
      Unit kb kb kb

    • report (репорт) – собрание измерений сенсора метрик, перечисленных в некотором промпте. Длительность сна сенсора перед очередным сбором измерений, формированием репорта и его отправкой определяется полем interval промпта.

    • specif – технические характеристики машины, на которой запущен сенсор. Используется в качестве справки в веб-приложении, а также для расчета показателей, выраженных в процентах.

    Модели данных

    Далее приводится описание и примеры моделей данных, фигурирующих во взаимодействии между клиентами, сервером и сенсорами. Использование одинаковых, четко регламентированных моделей данных во всех компонентах делает проще обеспечение корректности работы всей системы и поддержку кодовой базы.

    Описание моделей данных приводится в псевдокоде, но оно легко переносится и в классы Python, и в документы JSON (а значит, и в объекты JavaScript). Конкретные примеры объектов данных приводятся в формате JSON, посколько это предпочтительный способ взаимодействия компонентов системы.

    Metric

    Описание модели CpuMetric:

    Строковый литерал со следующими возможными значениями:

    system user nice idle iowait irq softirq steal guest guest_nice freq

    Описание модели NetMetric:

    Строковый литерал со следующими возможными значениями:

    recv sent errin errout dropin dropout

    Описание модели MemMetric:

    Строковый литерал со следующими возможными значениями:

    used cached shared swap

    Описание модели DskMetric:

    Строковый литерал со следующими возможными значениями:

    used read write

    Prompt

    Описание модели Prompt: 
    Prompt:
    # длительность сна сенсора перед очередным сбором измерений, формированием репорта и его отправкой
    # по умолчанию: 1
    interval: int

    # метрики, относящиеся к загруженности процессора
    cpu:
        # метрики, показатели которых будет собирать сенсор
        # по умолчанию: []
        metrics: list[CpuMetric]

        # будет ли сенсор собирать показатели по каждому ядру отдельно или агрегировать по всем
        # по умолчанию: false
        per_core: bool

    # метрики, относящиеся к сетевому трафику
    net:
        # метрики, показатели которых будет собирать сенсор
        # по умолчанию: []
        metrics: list[NetMetric]

        # будет ли сенсор собирать показатели по каждой сетевой карте отдельно или агрегировать по всем
        # по умолчанию: false
        per_card: bool

    # метрики, относящиеся к использованию оперативной памяти
    mem:
        # метрики, показатели которых будет собирать сенсор
        # по умолчанию: []
        metrics: list[MemMetric]

    # метрики, относящиеся к использованию постоянной памяти
    dsk:
        # метрики, показатели которых будет собирать сенсор
        # по умолчанию: []
        metrics: list[DskMetric]

        # будет ли сенсор собирать показатели по каждому диску и по каждому разделу отдельно или агрегировать по всем дискам
        # по умолчанию: false
        per_disk: bool
    Пример объекта Prompt: 
    {
  "interval": 2,
  "cpu": {
    "metrics": ["system", "user", "iowait", "freq"],
    "per_core": true
  },
  "net": {
    "metrics": ["recv", "sent"],
    "per_card": 1
  },
  "mem": {
    "metrics": ["used", "swap"]
  },
  "dsk": {
    "metrics": ["read", "write"],
    "per_disk": "false"
  }
}

    Булевы значения могут представляться как true, False, 1, "0", "T" и т.д. и будут корректно распарсены (сервером и сенсорами), но лучше использовать 0 и 1 как самые lightweight.

    Report

    Описание модели Report: 
    Report:
    # измерения загруженности процессора (по всему процессору или отдельно по каждому ядру)
    cpu: dict[CpuMetric, float] | list[dict[CpuMetric, float]]

    # измерения сетевого трафика (по всем сетевым картам или отдельно по каждой)
    net: dict[NetMetric, float] | list[dict[NetMetric, float]]

    # измерения использования оперативной памяти
    mem: dict[MemMetric, float]

    # измерения использования постоянной памяти (по всем дискам или отдельно по каждому разделу каждого диска)
    dsk: dict[DskMetric, float] | list[dict[DskMetric, float]]

    Specif

    Описание модели Specif: 
    Specif:
    # характеристики процессора
    cpu:
        # количество физических и логических ядер
        cores:
            phys: int
            logic: int

        # минимальная и максимальная частота каждого физического ядра
        freq: list[[int, int]]

    # характеристики сетевых карт
    net:
        # названия обнаруженных сетевых карт
        cards: list[str]

    # характеристики оперативной памяти
    mem:
        # объем оперативной памяти (исключая swap)
        size: int
        # объем swap-памяти
        swap: int

    # характеристики постоянной памяти
    dsk:
        # список разделов диска
        disks: list[]:
            # точка монтирования раздела
            mount: str
            # файловая система раздела
            fstype: str
            # опции раздела
            opts: str
            # объем дискового пространства раздела
            size: int

    Взаимодействие компонентов

    Здесь будет описание соединения

    С учетом этого можно уточнить схему взаимодействия сенсоров, сервера и клиентов:

    Клиентский API сервера

    • GET: sensor/group/list

      Описание:

      Получить список всех групп сенсоров, известных серверу, имеющих по крайнем мере один сенсор

      Возвращает:

      [
    {
        "sensor_id": <sensor_id>,
        "sensor_label": <sensor_label>,
        "sensor_group": <sensor_group>
    },
    ...
]

    • GET: sensor/list/all

      Описание:

      Получить список всех сенсоров, которые имеют активное подключение к серверу, независимо от группы

      Возвращает:

      [
    {
        "sensor_id": <sensor_id>,
        "sensor_label": <sensor_label>,
        "sensor_group": <sensor_group>
    },
    ...
]

    • GET: sensor/list/group

      Описание:

      Получить список всех сенсоров, которые имеют активное подключение к серверу, принадлежащих заданной группе

      Параметры:

      • group – заданная группа сенсоров

      Возвращает:

      [
    {
        "sensor_id": <sensor_id>,
        "sensor_label": <sensor_label>,
        "sensor_group": <sensor_group>
    },
    ...
]

    • GET: sensor/specif

      Описание:

      Получить технические характеристики машины, на которой запущен сенсор

      Параметры:

      • sensor-id – идентификатор сенсора

      Возвращает:

      Объект Specif в формате JSON

    • PUT: sensor/group/rename

      Описание:

      Переименовать группу сенсоров (переместить все сенсоры из заданной группы в другую)

      Тело:

      {
    "old_group": <old_sensor_group>,
    "new_group": <new_sensor_group>
}

      Возвращает:

      Пустое тело ответа в случае успешного выполнения, или сообщение об ошибке

      {
    "err": "Error message"
}

    • PUT: sensor/rename

      Описание:

      Переименовать сенсор (изменить его метку и/или группу). Если группы не существует, то она создается. При этом должен сохраняться инвариант, что метка уникальна внутри заданной группы, иначе запрос не выполнится с ошибкой

      Тело:

      {
    "sensor_id": <sensor_id>,
    "new_label": <new_sensor_label>
    "new_group": <new_sensor_group>
}

      Возвращает:

      Пустое тело ответа в случае успешного выполнения, или сообщение об ошибке

      {
    "err": "Error message"
}

    • GET: prompt

      Описание:

      Получить промпт, один из четырех: стандартный, расширенный, суточный и аналитический

      Параметры:

      • monitoring – вид мониторинга (standard, extended, intraday или analytic)

      Возвращает:

      Объект RichPrompt в формате JSON

    • GET: prompt/form

      Описание:

      Получить форму редактирования промпта: список всех доступных полей, соответствующие им доступные единицы измерения, а также словесные пояснения к ним

      Возвращает:

      Объект RichPrompt в формате JSON

    • PUT: prompt/edit

      Описание:

      Отредактировать промпт

      Параметры:

      • monitoring – вид мониторинга (standard, extended, intraday или analytic)

      Тело:

      Объект RichPrompt в формате JSON

      Возвращает:

      Пустое тело ответа в случае успешного выполнения, или сообщение об ошибке

      {
    "err": "Error message"
}

    • GET: history

      Описание:

      Получить все репорты заданного сенсора начиная с некоторого момента времени (в формате unix timestamp), но не более чем за последние 24 часа

      Параметры:

      • from – момент времени, начиная с которого запрашивается история репортов
      • sensor-id – идентификатор сенсора

      Возвращает:

      Массив объектов RichReport в формате JSON

      [
    <RichReport>,
    ...
]

    • WS: sub/standard

      Описание:

      Начать стандартный мониторинг в выделенном веб-сокете

      Принимает:

      Сразу после открытия соединения ожидает название группы сенсоров, по которой запрашивается мониторинг

      Возвращает:

      Массив объектов RichReport в формате JSON через фиксированные интервалы времени

      [
    <RichReport>,
    ...
]

    • WS: sub/extended

      Описание:

      Начать расширенный мониторинг в выделенном веб-сокете

      Принимает:

      Сразу после открытия соединения ожидает идентификатор сенсора, по которому запрашивается мониторинг

      Возвращает:

      Объект RichReport в формате JSON через фиксированные интервалы времени

    • WS: sub/intraday

      Описание:

      Начать суточный мониторинг в выделенном веб-сокете

      Принимает:

      Сразу после открытия соединения ожидает идентификатор сенсора, по которому запрашивается мониторинг

      Возвращает:

      Объект RichReport в формате JSON через фиксированные интервалы времени

    • WS: sub/analytic

      Описание:

      Начать аналитический мониторинг в выделенном веб-сокете

      Возвращает:

      Объект RichReport в формате JSON через фиксированные интервалы времени
